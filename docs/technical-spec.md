# ТЗ: Бот корпоративного справочника для express.ms

**Статус:** Черновик для уточнения  
**Платформа:** express.ms, BotX API  
**Внешняя система:** Active Directory через LDAPS  
**Целевая аудитория:** все корпоративные пользователи  
**Репозиторий:** https://github.com/v-bondarev/adsearch-express.git

---

## 1. Назначение бота

Бот предназначен для поиска сотрудников и контактов в Active Directory и выдачи карточки сотрудника с контактными данными в express.ms.

Пользователь вводит поисковый запрос в свободной форме. Бот ищет совпадения среди объектов AD типа "пользователь" и "контакт" по ФИО и частичным вариантам имени:

- ФИО
- ФИ
- ИО
- имя
- фамилия
- часть фамилии

Поиск выполняется только по русским и английским ФИО. Поиск по email, логину, телефону и другим идентификаторам не входит в MVP.

Состав полей карточки сотрудника должен быть определен после тестового LDAPS-запроса и получения полного набора доступных атрибутов.

### 1.1. Функциональные возможности бота

| № | Сценарий | Входные данные | Результат |
|---|---|---|---|
| 1 | Поиск сотрудника по полному ФИО | Текстовый запрос с ФИО | Если найден один сотрудник, бот возвращает результат поиска с кнопкой "Контакты" |
| 2 | Поиск сотрудника по фамилии и имени | Текстовый запрос с ФИ | Бот возвращает один или несколько найденных результатов |
| 3 | Поиск сотрудника по имени и отчеству | Текстовый запрос с ИО | Бот возвращает один или несколько найденных результатов |
| 4 | Поиск сотрудника по имени | Текстовый запрос с именем | Бот возвращает список совпадений, если они есть |
| 5 | Поиск сотрудника по фамилии или части фамилии | Текстовый запрос с фамилией или частью фамилии | Бот возвращает список совпадений, если они есть |
| 6 | Получение карточки сотрудника | Нажатие кнопки "Контакты" у найденного результата | Бот возвращает карточку сотрудника с контактными данными |
| 7 | Нет результатов | Любой поисковый запрос без совпадений | Бот сообщает, что сотрудник не найден |
| 8 | Несколько результатов | Поисковый запрос, которому соответствует несколько сотрудников/контактов | Бот возвращает несколько результатов, у каждого результата есть кнопка "Контакты" |
| 9 | Слишком много результатов | Поисковый запрос, которому соответствует больше 5 сотрудников/контактов | Бот показывает первые 5 результатов и предлагает уточнить запрос более полными данными |
| 10 | Точное совпадение | Поисковый запрос точно совпадает с ФИО сотрудника/контакта | Бот выводит только точное совпадение, без дополнительных частичных результатов |

**Команды бота:**

| Команда | Описание | Обязательна в MVP? |
|---|---|---|
| `/start` | Краткое приветствие и предложение ввести ФИО/часть ФИО для поиска | Да |
| `/help` | Подсказка по форматам поиска | Да |
| `/clear_cache` | Административная команда для ручного сброса кеша карточек | Да |

**Кнопки / интерактивные элементы:**

| Кнопка | Действие | Обязательна в MVP? |
|---|---|---|
| `Контакты` | Загружает и показывает карточку выбранного сотрудника | Да |

### 1.2. Что явно вне скоупа MVP

Предварительно вне скоупа, если не будет уточнено отдельно:

- редактирование данных AD из бота;
- создание, блокировка или изменение учетных записей;
- поиск вне Active Directory;
- поиск по email, логину, телефону и другим идентификаторам;
- массовая выгрузка справочника;
- административная панель;
- произвольные LDAP-запросы от пользователя;
- дополнительная авторизация пользователей через AD сверх ограничений express.ms.

## 2. Этапность разработки

### Этап 0 — Исследование AD и express.ms

До разработки необходимо выполнить реальные проверки:

1. Получить тестовый доступ к LDAPS.
2. Выполнить реальные LDAPS-запросы по сценариям поиска:
   - ФИО;
   - ФИ;
   - ИО;
   - имя;
   - фамилия;
   - часть фамилии;
   - отсутствие результатов;
   - несколько результатов;
   - объект пользователя;
   - объект контакта.
3. Получить полный список атрибутов для найденных тестовых объектов.
4. Определить финальный набор полей карточки сотрудника.
5. Зафиксировать реальные LDAP-фильтры, base DN, scope поиска и ограничения выдачи.
6. Проверить реальную структуру webhook от express.ms на тестовой регистрации бота.
7. Проверить формат авторизации исходящих запросов Bot -> BotX для конкретной версии express.ms.

Результат этапа: документ с подтвержденными примерами LDAPS-запросов/ответов, структурой webhook express.ms и финальным составом карточки.

### Этап 1 — Финальное ТЗ на разработку

После Этапа 0 в ТЗ должны быть зафиксированы:

- точные LDAP-фильтры;
- список AD-атрибутов для поиска;
- список AD-атрибутов для карточки;
- правила сортировки и ограничение выдачи до 5 результатов;
- точные тексты сообщений бота;
- структура кеша на 24 часа;
- правила доступа к административной команде `/clear_cache`;
- список администраторов бота через переменную окружения;
- параметры Docker Compose;
- список переменных окружения;
- схема локальной разработки и переноса на сервер.

## 4. Active Directory / LDAPS

### 4.1. Источник данных

Бот подключается к Active Directory по LDAPS.

Нужно уточнить:

- LDAPS URL;
- порт;
- base DN;
- bind DN / формат учетной записи для подключения;
- способ хранения секрета bind-пользователя;
- требуется ли клиентский сертификат;
- как проверяется TLS-сертификат AD;
- какие OU включать в поиск;
- какие OU исключать из поиска;
- какие классы объектов искать: users, contacts, оба типа подтверждены для MVP;
- как отличать активных пользователей от отключенных;
- как исключать сервисные, технические и скрытые учетные записи;
- какой атрибут фото использовать, если фото есть в AD.

### 4.2. Поиск

Поиск должен поддерживать запросы по:

- ФИО;
- ФИ;
- ИО;
- имени;
- фамилии;
- части фамилии.

Финальные LDAP-фильтры определяются после Этапа 0.

Предварительные требования к поведению:

- если найден один или несколько результатов, бот показывает список совпадений;
- в списке найденных результатов бот показывает ФИО, должность и подразделение;
- у каждого найденного результата есть кнопка "Контакты";
- по кнопке бот показывает карточку сотрудника;
- если найдено точное совпадение по ФИО, бот показывает только точное совпадение и не выводит дополнительные частичные совпадения;
- если пользователь и контакт имеют одинаковое ФИО, бот показывает оба результата отдельными строками;
- если результатов нет, бот сообщает об отсутствии совпадений;
- если найдено больше 5 результатов, бот показывает первые 5 и предлагает написать новый запрос с указанием более полных данных;
- карточки сотрудников кешируются на 24 часа;
- язык сообщений бота: русский.

## 5. Кеширование

Боту требуется кеш карточек сотрудников на 24 часа.

Требования:

- кешируются карточки сотрудников, результаты поиска не кешируются;
- хранилище кеша: SQLite;
- режим работы: один экземпляр бота в Docker Compose;
- если AD временно недоступен, но карточка сотрудника есть в кеше, бот может показать кешированную карточку;
- при показе кешированной карточки из-за недоступности AD бот должен явно написать: поиск временно недоступен, показан возможно устаревший результат;
- TTL кеша: 24 часа;
- должна быть административная команда `/clear_cache` для ручного сброса кеша;
- `/clear_cache` очищает весь кеш целиком;
- список администраторов, которым доступна команда `/clear_cache`, задается через переменную окружения.

Предварительная переменная окружения:

| Переменная | Назначение |
|---|---|
| `BOT_ADMIN_HUIDS` | Список HUID администраторов express.ms через запятую |

## 6. Фото сотрудников

В MVP нужно показывать фото сотрудника, если оно есть в AD.

Требования:

- источник фото: AD;
- точный атрибут фото определяется на Этапе 0 после тестового LDAPS-запроса;
- если фото отсутствует или не может быть обработано, карточка показывается без фото;
- если у объекта user и contact одинаковое ФИО, фото и карточка показываются для конкретного выбранного объекта, без объединения записей.

Нужно уточнить:

- какой атрибут используется в AD: `thumbnailPhoto`, `jpegPhoto` или другой;
- какие ограничения express.ms/BotX API есть на отправку изображений;
- нужен ли ресайз/сжатие фото перед отправкой.

## 7. Архитектура и код

Предварительная структура проекта:

```text
app/
├── main.py
├── config.py
├── db.py
├── ldap_client.py
├── botx_client.py
├── formatter.py
└── cache.py
```

Технические требования:

- стек разработки: Python, FastAPI, ldap3, httpx, SQLite;
- запуск через Docker Compose;
- сервер FastAPI/uvicorn;
- `/health` endpoint;
- секреты только через `.env`, без коммита в Git;
- обработка сетевых ошибок LDAPS;
- честные сообщения пользователю при ошибках: не найдено, слишком много результатов, временная недоступность AD, внутренняя ошибка;
- подробное debug-логирование raw webhook только на этапе исследования, без постоянного логирования персональных данных в production;
- все пользовательские сообщения только на русском языке;
- доступ обычных пользователей ограничивается средствами express.ms;
- дополнительная проверка обычного пользователя в AD не требуется;
- команда `/clear_cache` должна быть доступна только администраторам бота;
- логи не должны содержать текст поисковых запросов пользователей;
- в production-логах допускаются только технические события: запуск, остановка, ошибки подключения, статусы обработанных webhook, счетчики результатов без персональных данных.

## 8. Карточка сотрудника

Предварительный состав карточки сотрудника:

| Поле | Показывать в MVP? | Примечание |
|---|---|---|
| ФИО | Да | Обязательное поле для карточки |
| Должность | Да | Скрывать, если значение пустое |
| Подразделение | Да | Скрывать, если значение пустое |
| Рабочий телефон | Да | Скрывать, если значение пустое |
| Мобильный телефон | Да | Скрывать, если значение пустое |
| Email | Да | Скрывать, если значение пустое |
| Кабинет / офис | Да | Точный AD-атрибут определить по результатам тестовой выборки |
| Руководитель | Да | Скрывать, если значение пустое |

## 9. Разработка и перенос

Разработка выполняется локально, затем переносится на сервер через GitHub-репозиторий.

Репозиторий проекта: https://github.com/v-bondarev/adsearch-express.git

Требования:

- локальная разработка ведется в отдельной ветке;
- секреты и реальные параметры окружения хранятся только в `.env`, файл не коммитится;
- перед переносом на сервер код должен быть закоммичен и запушен в GitHub;
- на сервере код обновляется через `git pull`;
- на сервере локальные изменения кода не выполняются;
- после переноса выполняется пересборка Docker Compose и проверка `/health`;
- первый запуск с реальным express.ms и AD выполняется после проверки webhook и LDAPS на тестовых запросах.

## 12. Что нужно предоставить для Этапа 0

Секреты, пароли и реальные ключи не нужно отправлять в чат. Их нужно будет передать разработчику через согласованный безопасный канал или разместить в `.env` на тестовом стенде.

### От администратора AD

Нужно подготовить:

- LDAPS host и порт;
- base DN для поиска;
- список OU, которые включаются в поиск;
- список OU, которые исключаются из поиска;
- формат bind-пользователя: DN, UPN или `DOMAIN\user`;
- тестовую учетную запись для bind с правами только на чтение нужных атрибутов;
- способ проверки TLS-сертификата LDAPS: доверенный корпоративный CA или отдельный CA-файл;
- правила определения отключенных пользователей;
- правила определения сервисных/технических/скрытых учетных записей;
- соответствие AD-атрибутов полям карточки, если оно известно заранее;
- атрибут, в котором хранится фото сотрудника;
- 2-3 тестовых ФИО сотрудников/контактов, на которых можно проверять выдачу;
- разрешение на получение полного списка атрибутов для тестовых объектов.

### От администратора express.ms

Нужно подготовить:

- версию BotX API / ссылку на актуальную документацию именно для вашей установки;
- BOT_ID;
- способ получения BOT_SECRET_KEY или размещения секрета на стенде;
- webhook URL, который нужно зарегистрировать;
- сведения о доступности webhook: внутренний адрес или внешний HTTPS;
- HUID пользователей, которым разрешена команда `/clear_cache`;
- ограничения BotX API на отправку изображений;
- пример тестового сообщения боту для проверки webhook.
